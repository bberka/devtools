# Contributing to DevTools Collection

Thanks for helping improve DevTools Collection. This guide reflects the current Next.js + React implementation.

## Prerequisites

- Node.js 20 or newer
- npm
- Git
- Working knowledge of TypeScript, React, Tailwind CSS, and browser APIs

## Development Setup

```bash
npm install
npm run dev
```

The dev server runs at `http://localhost:3000` by default.

To verify a production static export:

```bash
npm run build
```

The generated static site is written to `out/`.

## Development Workflow

1. Create a feature branch.
2. Make focused changes that follow the existing patterns.
3. Run `npm run build`.
4. Manually test the affected tool or workflow in `npm run dev`.
5. Open a pull request with a clear summary and testing notes.

Commit prefixes used in this repo:

- `feat:` new features
- `fix:` bug fixes
- `docs:` documentation changes
- `refactor:` code restructuring without behavior changes
- `perf:` performance improvements
- `test:` tests
- `chore:` maintenance

## Coding Standards

- Use TypeScript for new code.
- Avoid `any` unless the API is genuinely dynamic and a narrower type would be misleading.
- Use React function components and hooks.
- Use existing UI components from `src/components/ui` before adding new primitives.
- Keep browser-only APIs inside client components, effects, or event handlers.
- Keep tools client-side and privacy-first unless a feature explicitly requires an external API.
- Prefer real-time processing for lightweight conversions.
- Use explicit action buttons for expensive operations, file uploads, cryptographic work, network requests, and exports.

## Adding a New Tool

### 1. Add Metadata

Edit `src/lib/utils/tools-config.ts`:

```ts
{
  id: 'example-tool',
  name: 'Example Tool',
  description: 'Do one useful developer task',
  category: 'utilities',
  icon: 'Wrench',
  keywords: ['example', 'utility'],
}
```

### 2. Create the Component

Create `src/components/tools/ExampleTool.tsx`:

```tsx
'use client';

import { useState } from 'react';
import { Card, CardContent, CardHeader, CardTitle } from '@/components/ui/card';
import { Textarea } from '@/components/ui/textarea';

export function ExampleTool() {
  const [input, setInput] = useState('');

  return (
    <Card>
      <CardHeader>
        <CardTitle>Example Tool</CardTitle>
      </CardHeader>
      <CardContent>
        <Textarea value={input} onChange={(event) => setInput(event.target.value)} />
      </CardContent>
    </Card>
  );
}
```

### 3. Wire the Renderer

Edit `src/components/ToolComponentRenderer.tsx`:

```tsx
import { ExampleTool } from '@/components/tools/ExampleTool';

const TOOL_COMPONENTS = {
  // existing tools
  'example-tool': ExampleTool,
} as const;
```

### 4. Verify

- Visit `/tools/example-tool` in dev.
- Confirm search, category filtering, favorites, recents, and command palette navigation work.
- Run `npm run build` and confirm the route appears in the static export.

## Tool UX Rules

- Use real-time updates for lightweight converters, formatters, encoders, and text transforms.
- Use buttons for expensive or explicit operations such as image processing, PDF/image export, password hashing, key generation, DNS lookups, and IP lookups.
- Every tool should have clear empty, error, loading, and success states where applicable.
- Include copy and clear actions when the output is user-facing text.
- Keep mobile layouts usable without horizontal overflow.
- Preserve dark mode styling.

## Static Export Rules

- Keep route data static-safe.
- Add tool routes through `TOOLS`; `generateStaticParams()` reads from that config.
- Do not add server-only behavior that breaks `output: 'export'`.
- External lookup tools should call public APIs from the browser and handle failures gracefully.

## Testing

Automated tests are not implemented yet. Until they are added, manually test:

- The affected tool's main success and error paths
- Light and dark mode
- Mobile and desktop layouts
- Favorites and recent tools
- Command palette navigation
- `npm run build`

## Current Contribution Priorities

- Add automated tests for route coverage and core tool behavior
- Replace `TOOL_COMPONENTS` with a typed registry or dynamic loader
- Improve bundle splitting for heavy tool dependencies
- Add PWA/offline support
- Add generated Open Graph images
- Decide whether the password hashing tool should add Argon2 support or stay bcrypt-only
- Add CSS tools such as glassmorphism, box-shadow, clamp calculator, and color picker

## Pull Request Checklist

- [ ] `npm run build` passes
- [ ] Affected workflow was manually tested
- [ ] UI works in light and dark mode
- [ ] Mobile layout was checked
- [ ] Documentation was updated when behavior changed
- [ ] No unrelated generated files were included
